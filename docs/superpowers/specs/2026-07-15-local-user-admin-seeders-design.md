# Локальные сидеры пользователя и администратора

## Цель

Добавить два идемпотентных Laravel-сидера для локальной разработки: обычного пользователя и администратора. Документировать запуск и локальные данные входа в `README.md`.

## Учётные записи

- Администратор: `admin@example.com` / `password`.
- Пользователь: `user@example.com` / `password`.

Обе записи создаются с подтверждённым email. Пароли передаются модели как обычная строка и хешируются существующим cast `hashed` модели `User`; открытый пароль не сохраняется в базе.

## Архитектура

- `Database\Seeders\UserSeeder` создаёт или обновляет обычного пользователя по email.
- `Database\Seeders\AdminSeeder` создаёт или обновляет администратора по email.
- `Database\Seeders\DatabaseSeeder` вызывает оба сидера, поэтому достаточно команды `php artisan db:seed`.
- Повторный запуск обновляет две целевые записи и не создаёт дубликаты.

В проекте нет поля роли или RBAC. Администратор получает права только когда его email присутствует в существующем allowlist `SEASONVAR_IMPORT_ADMIN_EMAILS`. Сидер не изменяет `.env` и не вводит новую модель авторизации.

## Документация

`README.md` получит раздел для локальных демонстрационных учётных записей с командой запуска, двумя парами email/password и требованием установить `SEASONVAR_IMPORT_ADMIN_EMAILS=admin@example.com` для административного доступа. Текст явно запретит использовать эти данные в production.

## Проверка

Feature-тест на SQLite in-memory должен подтвердить:

- создание обеих подтверждённых учётных записей;
- проверку обоих паролей через `Hash::check`;
- отсутствие дубликатов после повторного запуска `DatabaseSeeder`;
- доступ администратора через существующий gate при соответствующем значении `seasonvar.admin_emails`;
- отсутствие административного доступа у обычного пользователя.

После PHP-правок выполняются focused test, Pint для изменённых PHP-файлов и повторный focused test. Перед коммитом проверяется точный staged-состав, чтобы не включить посторонние изменения рабочего дерева.
