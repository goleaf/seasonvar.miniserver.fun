# Большие пакеты серий Seasonvar

## Проблема

`SeasonvarCatalogImporter::syncEpisodes()` выполняет один `Episode::upsert()` для всех серий страницы. Каждая строка содержит 13 bind-значений, а используемая SQLite собрана с `MAX_VARIABLE_NUMBER=32766`. Страница примерно с 2521 и более сериями поэтому завершается `too many SQL variables` и не может восстановиться повторным запуском.

## Решение

Импортёр сохраняет bulk-поведение, но делит изменившиеся серии на детерминированные пакеты по 50 строк. Такой пакет использует не более 650 bind-значений и остаётся безопасным даже для SQLite-сборок с консервативным лимитом 999.

Предварительная загрузка существующих серий не должна строить `whereIn(number)` по тысячам значений. Импортёр загружает regular-серии только затронутых сезонов и сопоставляет их в памяти по ключу `season_id|number`. Число сезонов одной страницы остаётся небольшим, а объём загруженных серий не превышает естественный размер синхронизируемого тайтла.

## Отклонённые варианты

- Построчный `updateOrCreate`: не достигает лимита SQLite, но создаёт тысячи отдельных запросов.
- Вычисление размера пакета из текущего driver limit: добавляет driver-specific ветвление и не защищает перенос базы на SQLite с меньшим compile-time limit.

## Проверка

Feature-тест импортирует страницу с 2600 сериями. До исправления он воспроизводит `too many SQL variables`; после исправления все 2600 записей сохраняются в одном сезоне. Existing importer tests подтверждают обычные страницы, сезоны, медиа и повторный импорт.

## Ограничения

- Публичной командой остаётся `php artisan seasonvar:import`.
- Видео не скачиваются; сохраняются только внешние ссылки и metadata.
- Новые dependencies и изменения схемы базы не нужны.
