# Полный безопасный прогрев публичного каталога

Дата: 2026-07-16

## Цель

Расширить `php artisan cache:warm-catalog` отдельным возобновляемым режимом, который последовательно прогревает все конечные канонические страницы анонимного публичного каталога. Существующий быстрый критический прогрев остаётся режимом по умолчанию и продолжает выполняться каждые десять минут.

Слово «все» в этом контракте означает все URL из явного безопасного публичного реестра, для которых ответ одинаков для любого гостя и не содержит чувствительных, авторизованных, подписанных или изменяющих состояние данных. Произвольные комбинации query-параметров не входят в конечный реестр.

## Рассмотренные подходы

### 1. Автоматически вызвать каждый `GET` route

Laravel route list можно обойти и попытаться автоматически подобрать параметры. Подход отвергнут: список содержит административные, авторизованные, signed, download, Livewire, health, auth и fallback-маршруты. Автоматическая классификация небезопасна, а значения динамических параметров нельзя корректно вывести из определения route.

### 2. Явный каталог безопасных целей с возобновляемым курсором

Для каждого разрешённого класса страниц задаётся отдельный источник канонических URL. Координатор обрабатывает небольшой пакет, сохраняет курсор и ставит следующий пакет в очередь. Это выбранный подход: он проверяем, не загружает весь каталог в память, позволяет продолжать многочасовой прогрев и по умолчанию запрещает все неизвестные маршруты.

### 3. Обход sitemap внешним crawler

Sitemap предоставляет значительную часть индексируемых страниц, но не описывает все страницы пагинации, API snapshots и эксплуатационные цели. Внешний crawler также хуже интегрируется с очередью, телеметрией и точечной инвалидизацией. Sitemap можно использовать как проверочный источник, но не как основной исполнитель.

## Проверенный объём

На момент проектирования приложение содержит:

- 199 определений маршрутов, включая 163 определения с `GET`;
- 32 926 опубликованных тайтлов;
- 48 478 сезонов;
- 724 507 серий;
- 873 562 медиазаписи;
- 111 257 используемых актёров и 21 698 используемых режиссёров;
- около 134 тысяч используемых элементов всех каталоговых справочников.

Полный прогрев не должен собирать эти записи одной коллекцией. Тайтлы, справочники, запросы и другие большие наборы обходятся по стабильному `id` через `chunkById` или эквивалентный keyset cursor.

## Классификация маршрутов

### Общий HTML-кэш и полный прогрев

В реестр входят только канонические гостевые страницы, защищённые `CachePublicPage` и политикой соответствующего профиля. Если безопасная страница из списка пока не имеет middleware, реализация сначала добавляет отдельный профиль политики и его тесты, а затем включает страницу в прогрев:

- главная `/` и поддерживаемые локализованные главные;
- `/titles` и все существующие страницы его пагинации без поиска;
- каждый опубликованный `/titles/{slug}` без выбора сезона, серии или медиаварианта;
- страницы годов и используемых каталоговых таксономий вместе с существующей пагинацией;
- индексы справочников и их существующая пагинация;
- `/stats`;
- стабильные индексируемые discovery-типы и их локализованные формы; случайные и персонализированные discovery-типы профиль отклоняет;
- `/collections` и его локализованный индекс без страниц конкретного пользователя;
- `/requests`, его локализованный индекс и существующая пагинация;
- публичные страницы заявок, уже допущенные профилем `requests`;

Redirect-маршруты справочников, legacy aliases и fallback не создают отдельную HTML-копию. Прогревается их каноническая конечная страница.

### Только внутренние snapshots или HTTP validators

Следующие маршруты могут безопасно вычисляться во время полного прохода, но их полные тела не помещаются в общий HTML-кэш:

- публичные read-only API endpoints;
- `sitemap*.xml`, `feed.xml`, `opensearch.xml` и `llms.txt`;
- статические публичные изображения и версии обложек.

Для них прогреваются существующие доменные snapshots или проверяется публичный ответ с соответствующим `Accept`. Потоковые XML-ответы и бинарные файлы не сохраняются как page-cache payload.

### Безусловный запрет

Никогда не включаются в общий кэш или полный прогрев:

- все `POST`, `PUT`, `PATCH`, `DELETE` и multi-method routes;
- `/admin/*` и routes с authorization gates;
- `/api/v1/me/*`, `/library*`, `/profile*`, `/settings*`, `/notifications` и другие account routes;
- страницы пользовательских проблем, личных заявок и приватных подборок;
- login, register, password reset, email verification и CSRF endpoints;
- signed playback routes и выдача/скачивание видео;
- Livewire assets, update, upload и preview routes;
- health/readiness endpoints и framework fallback;
- публичные профили пользователей, `/profiles/{userPublicId}/collections`, их локализованные формы и пользовательские media endpoints, потому что они относятся к персональным данным даже при публичной видимости; существующий `CachePublicPage` удаляется с profile collection routes;
- ответы с `Authorization`, активным пользователем, непубличными cookies или transient session state;
- поиск, произвольные фильтры и query-варианты `season`, `episode`, `media`, `variant`, `quality`, `format`.

Неизвестный новый route по умолчанию считается запрещённым. Для включения требуется явная регистрация и тест, доказывающий гостевую каноничность и отсутствие чувствительных данных.

## Интерфейс команды

Текущий вызов сохраняет поведение:

```bash
php artisan cache:warm-catalog
php artisan cache:warm-catalog --queue --refresh
```

Полный режим запускается явно:

```bash
php artisan cache:warm-catalog --scope=all-public --queue
```

Поддерживаются:

- `--scope=critical|all-public`, по умолчанию `critical`;
- `--dry-run` для подсчёта целей без HTTP-запросов и записей page cache;
- `--resume` для продолжения незавершённого поколения `all-public`;
- `--refresh` с прежним смыслом принудительного обновления внутренних snapshots.

Синхронный `all-public` запрещён из-за размера набора. Команда выводит русскую ошибку и рекомендует `--queue`. `--dry-run` разрешён синхронно.

## Архитектура

### Реестр безопасных целей

Отдельный сервис возвращает типизированные группы целей, а не произвольные URL. Цель содержит относительный канонический URL, ожидаемый тип ответа и профиль обработки. Самостоятельно переданный URL командой не принимается.

Источники целей разделены по ответственности:

- фиксированные публичные страницы;
- страницы пагинации каталога и справочников;
- опубликованные тайтлы;
- используемые годы и таксономии;
- стабильные discovery-типы;
- публичные заявки и другие уже разрешённые сущности;
- публичные документы и API snapshots без хранения чувствительного тела.

Каждый источник предоставляет детерминированный keyset cursor. Состояние не содержит Eloquent models, HTML, raw media URL, токены или cookie.

### Координатор и очередь

Полный прогрев использует отдельный job-класс, но существующую очередь `cache-warm-v2`, чтобы не требовать нового production worker. В очереди одновременно находится только один следующий пакет полного поколения, поэтому критический job может вклиниться между пакетами и сохраняет приоритет по времени постановки.

Один job:

1. получает атомарную блокировку поколения;
2. читает текущий segment и cursor;
3. обрабатывает ограниченный пакет или временной бюджет;
4. сохраняет прогресс и ограниченный список ошибок;
5. ставит следующий job, если цели остались;
6. помечает поколение завершённым, когда все источники исчерпаны.

Повторный обычный запуск создаёт новое поколение только при отсутствии активного. `--resume` продолжает последнее незавершённое поколение. Дубли job безопасны за счёт generation id, атомарного claim и `WithoutOverlapping`.

### HTTP-прогрев

Запросы выполняются только к настроенному same-origin адресу и сохраняют заголовок внутреннего прогрева. Redirect разрешается только внутри того же origin; внешний redirect считается ошибкой. Для каждого класса ответа задаётся правильный `Accept`.

Ошибка одной страницы не отменяет пакет. После ограниченных retry URL записывается как failed, курсор продвигается, а проход продолжается. В конце команда различает `completed` и `completed_with_failures`. Повторный `--resume` сначала повторяет ограниченный список ошибок, затем продолжает курсор.

## Ограничение нагрузки

- пакет и временной бюджет задаются через `config()`, а значения окружения читаются только в config-файле;
- начальная безопасная конфигурация использует один worker и последовательные same-origin запросы;
- каждый job завершается существенно раньше текущего 300-секундного worker timeout;
- между запросами допускается небольшой настраиваемый интервал;
- полный job не запускает параллельный import или внешние media downloads;
- при недоступности Redis, занятой generation lock или активной тяжёлой операции пакет откладывается с backoff;
- весь список URL никогда не материализуется в памяти или одном Redis value;
- отсутствие Redis `maxmemory` отражается предупреждением dry-run и в телеметрии, но не изменяется приложением автоматически.

TTL текущих cache profiles на первом выпуске не увеличивается. Поэтому полный проход гарантирует обработку каждой безопасной цели, но не обещает, что все страницы останутся одновременно resident после многочасового обхода: ранние записи могут перейти в stale или истечь. Цель первого прохода — измерить фактические bytes, duration и hit rate. Решение об увеличении TTL принимается только после измерения Redis и SQLite-нагрузки.

## Состояние и наблюдаемость

Состояние поколения хранится в operational Redis namespace с ограниченным retention и содержит:

- generation id и scope;
- status `queued`, `running`, `completed`, `completed_with_failures` или `failed`;
- текущий source и непрозрачный cursor;
- estimated, attempted, warmed, skipped и failed counts;
- started, updated и finished timestamps;
- duration, rate и приблизительный ETA;
- ограниченный список последних ошибок без URL с секретами;
- измеренный объём page-cache payload, если он доступен без полного scan.

`app:health` продолжает показывать состояние критического прогрева отдельно. Полный проход добавляется отдельным компонентом, чтобы многочасовой `running` не делал приложение degraded. Зависший cursor или отсутствие heartbeat дольше настроенного окна отмечается как degraded.

## Инвалидация и согласованность

Текущая точечная инвалидизация изменившихся тайтлов остаётся основной защитой свежести. Полный проход не заменяет её. Каждая цель строится на актуальной версии cache domain; если версия изменилась между claim и HTTP-ответом, следующий обычный запрос или targeted warm создаёт запись нового поколения.

Полный проход не очищает весь Redis, Memcached или page-cache namespace. Перезапуск выполняется созданием нового operational generation и существующими versioned keys.

## Тестирование

Обязательные PHPUnit-сценарии:

- прежний `critical` вызов и schedule не меняют поведение;
- `all-public` без `--queue` отклоняется, кроме `--dry-run`;
- dry-run не отправляет HTTP и не записывает page payload;
- генераторы обходят большие таблицы порциями и создают только канонические URL;
- все текущие маршруты классифицированы, а новый неизвестный route проваливает contract test до явного решения;
- административные, авторизованные, signed, download, Livewire, health, auth и account routes отсутствуют в allowlist;
- title targets не содержат query выбора серии или media;
- job продолжает пакет после HTTP-ошибки и сохраняет безопасную диагностику;
- дубликаты job не обрабатывают один cursor дважды;
- `--resume` продолжает незавершённое поколение;
- full warm не раскрывает raw media URL, token, cookie или private identifiers;
- localized и canonical URL не дублируются;
- все внешние HTTP-запросы запрещены через `Http::preventStrayRequests()`.

После focused tests запускаются Pint и полный релевантный набор cache/route tests. Frontend build не требуется, если Blade, CSS, JS и Vite-конфигурация не меняются.

## Документация и выпуск

Реализация обновляет `docs/caching.md`, эксплуатационные инструкции из карты `docs/README.md`, `README.md` и `CHANGELOG.md`. Управляемые `project-docs` блоки изменяются только через `php artisan project:docs-refresh`.

Выпуск выполняется ступенчато:

1. dry-run и route safety audit;
2. ограниченная проверка первых 1 000 целей в тестовом окружении;
3. проверка ошибок, SQLite latency и Redis memory;
4. проход 10 000 целей;
5. полный проход;
6. решение о частоте повторного полного прогрева по фактическому времени и hit rate.

Автоматический десятиминутный запуск остаётся только для `critical`. Полный `all-public` запускается оператором после выпуска или крупного завершённого импорта.
