# Параллельный импорт Seasonvar через Redis

Дата: 12.07.2026

## Цель

Ускорить `php artisan seasonvar:import` до десяти параллельных обработчиков, не допуская повторного скачивания одной страницы разными workers и сохраняя регулярную проверку изменившихся данных источника: постера, описания, сезонов, серий и внешних вариантов воспроизведения.

Команда `seasonvar:import` остается единственной публичной командой импорта Seasonvar. Видео-файлы приложение не скачивает.

## Согласованный режим эксплуатации

- Cron запускает диспетчер импорта ровно 10 раз в сутки.
- Десять постоянно работающих Laravel queue workers читают отдельную Redis-очередь `seasonvar-import`.
- Повторный cron-запуск не ставит повторно уже поставленную или обрабатываемую страницу.
- Успешно импортированная страница по умолчанию снова становится кандидатом через 24 часа. Интервал остается настраиваемым через `SEASONVAR_IMPORT_REFRESH_AFTER_HOURS`.
- Страницы с недостающими данными повторяются через `SEASONVAR_IMPORT_MISSING_DATA_RETRY_HOURS`; HTTP-ошибки продолжают использовать существующий backoff.
- Десять запусков диспетчера в сутки не означают десять скачиваний каждой страницы. Они уменьшают задержку постановки новых и просроченных страниц в очередь.

## Рассмотренные варианты

### Redis-очередь и отдельные workers — выбранный вариант

Redis уже доступен на сервере, PHP-модуль Redis установлен, а Laravel имеет настроенное Redis-соединение. Очередь не пишет служебные операции в SQLite `jobs`, поэтому десять workers создают меньше конкурирующих записей в основной базе. Процесс-менеджер может перезапускать workers после сбоя или деплоя.

### Дочерние процессы внутри `seasonvar:import`

Laravel Concurrency или прямой запуск процессов дал бы параллельность в рамках одной команды, но усложнил бы восстановление после падения, повторные попытки, фоновый запуск и безопасную остановку. Родительский процесс оставался бы единой точкой отказа.

### Ручное деление каталога на десять диапазонов

Параметры shard/range сделали бы запуск простым, но потребовали бы стабильного разбиения изменяемого sitemap. Ошибка в диапазонах могла бы привести к пропускам или пересечениям, поэтому этот вариант не используется.

## Архитектура

### Режимы команды

Существующий синхронный режим сохраняется для импорта одной URL и для совместимости с текущими тестами и ручной диагностикой.

Для полного каталога команда получает явный queued-режим. В нем она:

1. Получает короткую coordinator-блокировку, не удерживая старую глобальную блокировку на все время обработки очереди.
2. Обновляет sitemap и выполняет безопасный `upsert` найденных URL.
3. Восстанавливает просроченные claims предыдущих запусков.
4. Выбирает только pending, failed-due, missing-data-due и stale страницы через существующий `SeasonvarRefreshPlanner`.
5. Атомарно закрепляет каждую страницу за текущим запуском.
6. Отправляет задание в Redis только после успешного закрепления.
7. Запускает асинхронный контроль завершения и сразу заканчивает CLI-процесс.

Параметр URL не отправляется в массовую очередь: адресная команда остается предсказуемой и синхронной.

### Lease/claim страницы

В `source_pages` добавляются nullable-поля идентификатора claim, времени получения, времени истечения и идентификатора import run. Для выборки и восстановления добавляются составные индексы.

Claim получается условным `UPDATE`: запись разрешена только если claim отсутствует или уже истек. Это условие является основным источником истины для защиты от дублей и работает даже при двух одновременно запущенных диспетчерах.

Если dispatch в Redis завершился ошибкой, только что полученный claim освобождается. Если worker умер, claim становится доступен после настраиваемого TTL. Старое задание с неверным token завершается до HTTP-запроса и не может скачать страницу повторно.

### Queue job

Одно задание обрабатывает одну `SourcePage` и несет только scalar ID, claim token, import run ID и флаг force. Задание:

- использует Redis-очередь `seasonvar-import`;
- проверяет token и срок claim до HTTP-запроса;
- имеет ограниченные attempts, timeout и экспоненциальный backoff;
- освобождает claim после зафиксированного успеха или окончательной ошибки;
- атомарно обновляет счетчики `SeasonvarImportRun`;
- реализует `failed()` с журналированием и освобождением собственного claim.

Дополнительная overlap-блокировка строится по стабильному external ID Seasonvar, извлеченному из URL. Поэтому разные сезоны одного тайтла не изменяют его связи одновременно. Если external ID получить нельзя, используется `url_hash`. Блокировка влияет только на одновременное выполнение: page claim продолжает предотвращать повторную постановку и повторный HTTP-запрос.

### Проверка обновлений

Когда страница стала кандидатом по freshness policy, worker всегда выполняет HTTP-запрос к разрешенному URL `https://seasonvar.ru/`. Тело сравнивается с сохраненным `content_hash`.

- Если hash не изменился и связанные данные полны, обновляются crawl timestamps, а тяжелый HTML parse и database sync пропускаются.
- Если hash изменился, текущий parser повторно извлекает и синхронизирует метаданные, включая `poster_url`.
- Если прежние данные неполны или медиа требуют обновления, существующая логика может выполнить sync даже при неизменном hash.
- Внешние media URL проверяются существующими timeout/retry правилами; сами видео не загружаются.

HTTP client сохраняет timeout, connect timeout и retry. Crawl delay действует отдельно в каждом worker; максимальный параллелизм ограничен количеством workers — 10.

### Завершение запуска

Общие операции не выполняются в каждом page job. Уникальный finalizer периодически проверяет claims текущего запуска. Когда незавершенных claims не осталось, он один раз выполняет storage maintenance, нормализацию/очистку связей, объединение тайтлов, media backlog и перестроение рекомендаций, затем переводит run в `completed` или `failed`.

Запуск без новых кандидатов завершается сразу и не создает пустую очередь. Просроченные или потерянные claims отражаются в summary запуска и возвращаются планировщику без повторной обработки живым worker.

## Состояния и совместимость

В `seasonvar_import_runs` добавляется признак execution mode, чтобы синхронный Linux-процесс и асинхронная очередь различались явно. Существующее восстановление process lock применяется только к sync-run и не помечает живой queued-run ошибочным.

Текущие поля `parse_status`, `import_status`, `retry_after_at`, `last_crawled_at`, `last_imported_at`, `last_changed_at` и `content_hash` сохраняют доменное значение. Lease-поля отвечают только за конкурентное владение и не подменяют статусы импорта.

Все schema changes выполняются новой additive и обратимой миграцией. Старые миграции не редактируются.

## SQLite и Redis

Очередь импорта всегда задается как Redis независимо от общего `QUEUE_CONNECTION`, чтобы 10 workers не резервировали задания в SQLite. Основная каталоговая база остается SQLite с WAL и Laravel `busy_timeout`.

Транзакции каталога остаются короткими; внешние HTTP-запросы не переносятся внутрь транзакций. Ошибки `database is locked` проходят через ограниченные queue retries и существующие transaction attempts. Новая production dependency не добавляется.

## Фоновый запуск

Рекомендуемый production-вариант — systemd template unit с `User=www`, `WorkingDirectory=/www/wwwroot/seasonvar.miniserver.fun` и командой `/usr/bin/php artisan queue:work redis --queue=seasonvar-import`. Запускаются и включаются экземпляры `1..10`. Workers периодически перезапускаются через `--max-time`/`--max-jobs`, чтобы освобождать накопленную память.

После реализации документация должна содержать:

- команды миграции и проверки Redis;
- точную команду ручной постановки импорта;
- временную `nohup`-команду для запуска 10 workers без установки нового пакета;
- надежную systemd-конфигурацию;
- cron expression с десятью временами запуска в сутки;
- команды просмотра очереди, failed jobs, логов и штатного перезапуска workers.

## Cron

Точные часы запуска по серверному времени:

`00:00, 02:00, 05:00, 07:00, 10:00, 12:00, 14:00, 17:00, 19:00, 22:00`.

Соответствующий cron hour list: `0,2,5,7,10,12,14,17,19,22`. Каждый запуск выполняет queued-режим `seasonvar:import`; coordinator lock и claims делают пересекающиеся cron-процессы безопасными.

## Обработка ошибок

- Ошибка sitemap не удаляет ранее известные страницы и завершает coordinator run с диагностикой.
- Ошибка Redis dispatch освобождает claim страницы и учитывается в run.
- HTTP 404 и временные HTTP-ошибки сохраняют существующие статусы и retry policy.
- Worker с устаревшим или чужим token завершает job без HTTP-запроса.
- После исчерпания attempts `failed()` освобождает только claim с совпадающим token.
- Finalizer не выполняет общие операции параллельно для одного run.
- Очередной cron восстанавливает истекшие claims, но никогда не отбирает живой claim.

## Тестирование

Новые PHPUnit-тесты должны доказать:

1. queued-команда отправляет в Redis только успешно закрепленные eligible страницы;
2. второй dispatcher не ставит страницу с живым claim повторно;
3. истекший claim может быть безопасно восстановлен;
4. job с неверным token не выполняет HTTP-запрос;
5. разные страницы одного external ID не выполняются одновременно;
6. разные тайтлы могут обрабатываться независимо;
7. неизменный hash пропускает тяжелый sync, но обновляет время проверки;
8. изменившийся HTML обновляет `poster_url` и остальные parsed facts;
9. finalizer запускает общие операции один раз после исчезновения claims;
10. sync URL-import и текущие importer tests сохраняют поведение;
11. все внешние запросы тестов закрыты `Http::fake()` и `Http::preventStrayRequests()`.

После focused tests выполняются Pint и полный `php artisan test`. Изменений frontend assets нет, поэтому `npm run build` для этой задачи не требуется.

## Критерии готовности

- Десять workers одновременно обрабатывают разные страницы/тайтлы из Redis.
- Одна страница не скачивается двумя workers при повторном cron, retry или зависшем старом job.
- Сезоны одного тайтла не изменяют общие связи конкурентно.
- Свежие страницы не скачиваются при каждом из десяти cron-запусков.
- Просроченные страницы повторно запрашиваются и обновляют изменившийся постер и другие данные.
- После сбоя очередь автоматически продолжает работу без ручного удаления locks.
- Пользователь получает проверенные команды для ручного фонового запуска 10 workers, systemd и cron.
