# Seasonvar title-group queue lock

Дата: 12.07.2026

## Проблема

Очередь исключает повторную обработку одной `source_page` через атомарный claim, но lock worker строится по числовому идентификатору страницы Seasonvar. У разных сезонов одного сериала эти идентификаторы различаются. В результате несколько workers одновременно обновляют один `catalog_titles` и связанные записи, что вызывает `database is locked` в SQLite.

## Решение

`SeasonvarImportGroupKey` должен извлекать из serial URL стабильную часть slug до технического `_ps...` и/или суффикса сезона, привести регистр и удалить лишние ведущие разделители из вариантов с двойным дефисом после numeric ID. Это должно работать и для legacy URL без `_ps`, включая единственный production-вариант с `_1-season` вместо `-1-season`. Все страницы сезонов одного сериала получают одинаковый Redis lock key. Если URL не соответствует известному формату, используется существующий `url_hash` fallback.

Worker должен пересчитывать ключ из текущей `SourcePage` перед получением Redis lock. Сериализованный `groupKey` сохраняется для совместимости и диагностики, но не является источником истины: уже находящиеся в Redis jobs были созданы со старым numeric key.

Canonical lock устраняет конфликт сезонов одного тайтла, но SQLite допускает только одного writer даже для разных тайтлов. SQLite connection должна использовать `IMMEDIATE` transaction mode вместе с уже настроенными WAL и `busy_timeout=10000`: writer lock запрашивается в начале transaction, а не через проблемный DEFERRED read-to-write upgrade. Внешние HTTP-запросы остаются вне transaction и продолжают выполняться десятью workers параллельно.

Числовой идентификатор страницы не используется как title-group key: он идентифицирует конкретную страницу сезона, а не общий тайтл.

## Границы

- DB claims остаются источником истины для уникальности конкретной страницы.
- Параллельность сохраняется между разными сериалами.
- Видео не скачиваются; меняется только координация parser jobs.
- Схема базы данных и публичная команда `seasonvar:import` не меняются.

## Проверка

- Разные URL сезонов одного canonical slug дают одинаковый ключ.
- Разные сериалы дают разные ключи.
- Неизвестный формат использует hash fallback.
- Job со старым сериализованным ключом блокируется новым canonical key, вычисленным во время обработки.
- SQLite connection использует `IMMEDIATE` transaction mode.
- Existing queued import tests и полный PHPUnit suite остаются зелёными.
