# Strict media backlog format design

Date: 2026-07-16

## Evidence and problem

`LicensedMediaFileSizeBacklog::eligibleQuery()` currently accepts a row when either its normalized `licensed_media.format` belongs to the configured direct-download formats or the complete effective URL contains a substring such as `.mp4`. The URL predicate is broader than the canonical `ExternalMediaFileType`, which derives URL format only from `parse_url(..., PHP_URL_PATH)` and never treats a query parameter as a trusted extension.

A read-only production audit found 873,561 HTTP(S) media rows with a stored direct format and exactly zero additional rows contributed by the URL-substring fallback. It also found zero direct-format rows containing an HLS marker, zero non-HTTP direct-format rows and zero non-direct rows containing a direct-extension marker. Both current and strict SQL plans scan `licensed_media`; the fallback therefore adds repeated wildcard comparisons without changing the result set.

## Considered approaches

1. Keep the fallback. This retains no observed compatibility and preserves a classification rule that is intentionally broader than inspection and download eligibility.
2. Parse URL paths inside SQL. SQLite, MySQL and PostgreSQL do not share a concise URL parser, so this would create a second database-specific classifier and still duplicate `ExternalMediaFileType`.
3. Treat normalized stored format as the only SQL preselection signal, keep the existing HTTP(S) effective-URL condition, and retain the canonical PHP type check immediately before any request. This is selected because importers already persist path-derived format, current production data loses no row, and defense in depth remains in place.

## Design

`LicensedMediaFileSizeBacklog::eligibleQuery()` will require `format IN (configured direct formats)` and an HTTP(S) effective URL. It will remove only the `OR effective_url LIKE %.format%` loop. Freshness, soft-delete scope, stable `lazyById`, limits, status aggregate, cache store/TTL and file-size state transitions remain unchanged. The operational cache resource advances from `v3` to `v4` so a snapshot computed under the broader predicate cannot survive the deployment's bounded stale window.

The SQL query is a bounded preselection boundary, not final authorization. `InspectLicensedMediaFileSize` continues to delegate to `ExternalMediaFileSizeInspector`, which calls `ExternalMediaFileType::isPlaylist()` before `isDirect()` and performs URL validation before networking. Download eligibility independently repeats the same type and URL checks. A malformed or contradictory stored row therefore remains unable to turn HLS into a direct download.

No migration or backfill is required. If a future importer creates a direct file with `format=stream|null`, that importer must normalize the path through the existing metadata boundary rather than teaching the backlog to infer extensions from arbitrary URL text.

## Data flow

```text
licensed_media stored format + effective HTTP(S) URL
    -> strict SQL backlog preselection
    -> ExternalMediaFileType playlist/direct classification
    -> ExternalMediaUrlGuard validation
    -> bounded HEAD / one-byte Range inspection
```

## Verification

No automated tests will be created or executed under the task constraint. Verification will compare current versus strict production counts, inspect the SQL generated by the Eloquent query, run PHP lint, path-targeted Pint, focused Larastan, `seasonvar:import --status`, `git diff --check` and task-only diff/security searches. The strict eligible total must remain 873,561 for the audited snapshot, and the query must contain no direct-extension wildcard predicate.
