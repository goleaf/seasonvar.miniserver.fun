# Русскоязычный README и пользовательская история обновлений

## Цель

Сделать `README.md` понятной русскоязычной точкой входа для посетителя и разработчика: описать актуальные возможности портала, последние новости, завершённые работы и дальнейшие планы, а в самом конце вести датированную историю изменений для посетителей без внутренней терминологии разработки.

## Язык

Весь обычный текст `README.md` пишется по-русски. Имена продуктов и технологий, команды, параметры, маршруты, имена файлов, переменных окружения, форматы данных и другие точные идентификаторы сохраняются в исходном написании, чтобы инструкции оставались исполнимыми.

Новая автоматическая проверка анализирует текст вне ограждённых блоков кода. Каждая содержательная строка должна содержать русские буквы; пустые строки, разделители Markdown и служебные HTML-комментарии разрешены. Это исключает полностью англоязычные заголовки и абзацы, но не запрещает точные технические имена внутри русского предложения.

## Структура README

Файл получает следующую последовательность разделов:

1. назначение проекта и состояние;
2. последние новости;
3. возможности посетителя;
4. полный свод уже реализованных направлений;
5. установка, запуск, импорт и эксплуатационные команды;
6. ссылки на тематическую документацию;
7. автоматически управляемый блок `project-docs`;
8. дорожная карта со статусами «в разработке» и «запланировано»;
9. история обновлений для посетителей — последний раздел файла.

Технические обращения и двуязычная главная, находящиеся в незавершённом рабочем дереве, отмечаются как функции в разработке. Опубликованные коммиты до текущего `HEAD` описываются как готовые. История рассказывает о результате для посетителя и не перечисляет классы, миграции, внутренние задачи или названия коммитов.

## Правило для агентов

В `AGENTS.md` добавляется обязательное правило:

- после каждого запроса перед завершением работы проверять актуальность `README.md`;
- при изменении пользовательских возможностей, состояния функции или дорожной карты обновлять соответствующий раздел и добавлять датированную запись в пользовательскую историю;
- не создавать пустую запись, если состояние продукта не изменилось;
- сохранять весь обычный текст `README.md` на русском языке;
- не редактировать вручную управляемый блок `project-docs`.

Такое правило охватывает запросы пользователя, которые Git не способен наблюдать напрямую.

## Git-hook

Новый скрипт `scripts/check-readme-policy.sh` проверяет файл из рабочего дерева или индексную версию перед коммитом. `.githooks/pre-commit` вызывает его после существующих проверок безопасности.

Hook выполняет три проверки:

1. в `README.md` присутствуют разделы «Дорожная карта» и «История обновлений для посетителей»;
2. пользовательская история является последним разделом второго уровня;
3. обычные текстовые строки русскоязычны.

Если коммит затрагивает код приложения, конфигурацию, миграции, развёртывание, переводы, публичные ресурсы, маршруты или зависимости, индекс обязан содержать изменение `README.md`. Документационные, тестовые и внутренние служебные коммиты не требуют искусственной записи. Автокоммит управляемой документации сохраняет существующий рекурсивный предохранитель.

## Проверки

- PHPUnit-тест запускает скрипт на корректном и намеренно ошибочных временных файлах.
- Тест проверяет отсутствие пользовательской истории, раздел после истории и полностью английскую строку.
- Выполняются `bash -n`, focused PHPUnit-тест, `php artisan project:docs-refresh --check`, проверка Markdown-ссылок через существующую команду и `git diff --check`.
- Так как изменяется PHP-тест, после правки запускается Pint для изменённых PHP-файлов.

## Git и незавершённое рабочее дерево

Коммит включает только изменения этой задачи: `README.md`, `AGENTS.md`, `docs/README.md`, `docs/development.md`, hook, проверочный скрипт, тест, спецификацию и план. Другие уже существующие staged и unstaged изменения сохраняются без редактирования. Перед отправкой проверяются ветка `main`, состав коммита и удалённый результат.
