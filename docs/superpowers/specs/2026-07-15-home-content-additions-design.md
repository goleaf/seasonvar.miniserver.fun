# Фактические обновления контента на главной

Дата: 15.07.2026

## Цель

Блок «Последние обновления» должен реагировать только на создание хотя бы одной публично доступной серии или видеозаписи. Изменение названия, описания, постера, рейтинга, связей, health-состояния или других метаданных не должно поднимать тайтл в этом блоке.

Блок «Новые серии» должен показывать один тайтл одной строкой-карточкой. Внутри карточки перечисляются все серии и видеоварианты, добавленные тайтлу в его последний календарный день пополнения. Номер сезона, номер и название серии, качество, перевод, формат и дата публикации видео остаются видимыми.

Новый тайтл попадает в оба блока только тогда, когда вместе с ним создана хотя бы одна серия или видеозапись.

## Рассмотренные подходы

1. Отдельный `content_updated_at` в `catalog_titles`. Выборка была бы дешёвой, но каждый путь создания серии или видео — importer, playlist importer и admin writes — обязан синхронно обновлять поле. Пропущенный вызов дал бы тихо неверную ленту.
2. Вычисление по `episodes.created_at` и `licensed_media.created_at`. Эти timestamps уже являются фактом создания строки, не меняются при metadata update и одинаково работают для всех существующих путей записи. Для главной результат кешируется, а новые составные индексы покрывают выборку дочерних строк.
3. Отдельный журнал событий добавления. Он дал бы точную import-batch историю, но добавил бы новую retention, backfill и transactional dual-write границу, которые не нужны двум блокам главной.

Выбран второй подход: фактические дочерние строки являются единственным источником истины, без нового mutable timestamp и без отдельного event store.

## Архитектура и поток данных

Новый focused query service в `App\Services\Catalog` строит объединение двух публичных наборов:

- созданные `Episode`, доступные вместе с их `Season` и `CatalogTitle`;
- созданные опубликованные `LicensedMedia`, доступные вместе с release hierarchy и `CatalogTitle`.

Объединение агрегируется по `catalog_title_id` с `MAX(created_at)`. Оно возвращает не более 48 тайтлов в порядке `added_at DESC, catalog_title_id DESC`. `CatalogHomeSnapshotCache` сохраняет ID и точный timestamp последнего фактического пополнения и меняет версию content-index cache key, поэтому старый snapshot по `indexed_at` после развёртывания не используется.

`CatalogHomePageBuilder` гидратирует карточки текущим visibility-aware query, назначает им presentation-only `content_added_at` и группирует «Последние обновления» по этой дате. `indexed_at` остаётся поисковым timestamp и больше не определяет этот блок.

Для «Новых серий» берутся первые 12 видимых тайтлов из того же списка. Для каждого тайтла query service одним запросом серий и одним запросом видео загружает все публичные additions за календарную дату его последнего пополнения. Результат группируется в PHP по тайтлу и передаётся Blade уже подготовленными коллекциями; запросов из Blade и N+1 нет.

Плоский `latestMedia` сохраняется для текущего API-контракта, но web-view использует новые сгруппированные данные.

## Представление

Одна `x-catalog.latest-media-card` соответствует одному тайтлу и выводит постер/название только один раз. Внутри неё:

- серия выводится один раз с сезоном, номером и необязательным названием;
- все новые видеоварианты этой серии перечисляются отдельными видимыми строками;
- видео для старой серии всё равно показывает сезон и серию через загруженную связь;
- новая серия без видео остаётся видимой с явным нейтральным сообщением об отсутствии видео;
- тайтловое видео без `episode_id` выводится отдельной строкой без выдуманного номера серии.

Каждый видеовариант сохраняет ссылку на конкретный `episode`/`media` и текущие подписи качества, перевода, формата и даты. Разметка остаётся светлой одноколоночной, mobile-first и использует существующий `x-ui.poster-card`.

## Производительность и индексы

Additive reversible migration добавляет:

- `episodes (season_id, created_at, id)` для загрузки добавлений тайтла через сезоны;
- `licensed_media (catalog_title_id, created_at, id)` для агрегации и загрузки новых видео.

Существующие одиночные индексы не удаляются. Тяжёлая агрегированная выборка выполняется только при rebuild кешированного snapshot; web request обычно читает snapshot и делает ограниченную гидратацию.

## Совместимость и ошибки

- Публичные publication/audience/availability scopes применяются к тайтлу, сезону, серии и media.
- Soft-deleted и недоступные строки не участвуют в лентах.
- Если cached row больше не видим, гидратация его отбрасывает; порядок остальных сохраняется.
- Пустой набор сохраняет существующие русские empty states.
- API не получает raw URLs или importer state; существующий `latest_releases` остаётся плоским.

## Проверки

Feature tests должны подтвердить:

1. более новый `indexed_at` после metadata-only изменения не поднимает тайтл выше тайтла с реально новой серией;
2. тайтл без серий и видео не попадает в «Последние обновления»;
3. создание видео для существующей серии считается фактическим обновлением;
4. несколько новых серий и несколько видео одного тайтла дают одну карточку тайтла, но сохраняют все номера серий и metadata каждого видео;
5. новая серия без видео остаётся видимой;
6. SQL использует стабильный tie-breaker, а migration создаёт ожидаемые составные индексы;
7. focused PHP tests, Pint, полный PHPUnit-набор, `npm run build` и `project:docs-refresh --check` проходят.
