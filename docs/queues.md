# Очереди и jobs

Обновлено: 16.07.2026

## Конфигурация

- Основное production подключение очереди по умолчанию: `QUEUE_CONNECTION=redis`, Redis connection `queues`. Database queue остаётся framework fallback для локальной диагностики, но не является production default.
- Таблицы `jobs`, `job_batches` и `failed_jobs` создаются миграцией `0001_01_01_000002_create_jobs_table.php`.
- Для database, Redis и Beanstalkd `retry_after` по умолчанию равен 1200 секундам. Это больше тайм-аута долгого job импорта и защищает от повторного запуска той же работы, пока worker еще ее выполняет.
- `composer dev` запускает локальный `queue:listen` с `--timeout=900 --tries=3`; локальный environment должен иметь Redis либо явно выбрать безопасный test driver.
- Основные очереди: `seasonvar-title-refresh`, `seasonvar-import`, `critical`, `cache-warm-v2`, `default`; историческая `cache-warm` изолирована из-за истёкших payload deadlines. Будущие `notifications`, `media-check`, `maintenance`, `low-priority` запускаются отдельными supervisors только когда появляются jobs и SLO.
- Cache-warm worker устанавливается из `deploy/systemd/seasonvar-cache-warm-worker.service` и слушает только `cache-warm-v2`; `cache-warm`, `critical` и `default` в этот unit не входят. Общий импорт слушает только `seasonvar-import` через `seasonvar-import-worker@.service`, а открытая пользователем карточка обрабатывается отдельным приоритетным пулом `seasonvar-title-refresh-worker@.service`. Для текущего четырёхъядерного SQLite-сервера production baseline — 4 import и 8 title-refresh процессов. Число известных страниц тайтла остаётся без application-level limit; baseline ограничивает только доступную одновременную IO capacity. Расширять пул можно только после измерения queue latency, CPU и SQLite writer contention; разные `catalog_title_id` остаются независимыми, а unique/lock одного тайтла не блокирует обновление остальных.
- Строго однопоточный профиль устанавливается из `seasonvar-import-forever.service` и не запускает queue worker. Он взаимоисключающий с import/title-refresh worker templates и queued cron: одновременно выбирается только один профиль. В однопоточном режиме browser-triggered title jobs не потребляются отдельно, а карточка видит изменения следующего полного sync-цикла через Livewire polling.
- Scheduler mutex для `withoutOverlapping`/`onOneServer` явно использует `redis-locks`; default domain cache не участвует в координации schedule.
- `seasonvar-media-file-size-backfill` каждые 10 минут вызывает только существующий `seasonvar:import --refresh-media-sizes --media-size-limit=N`. Default `N=20`; `SEASONVAR_MEDIA_FILE_SIZE_SCHEDULED_BACKFILL_ENABLED` отключает доставку, а `SEASONVAR_MEDIA_FILE_SIZE_SCHEDULED_BACKFILL_LIMIT` задаёт bounded batch. Scheduler lock (`withoutOverlapping(30)`, `onOneServer`) дополняет, но не заменяет общий importer/global-run single-flight, поэтому отдельной queue job или параллельного media importer нет.
- `WarmCatalogCaches` сохраняет legacy boolean `refresh` только для совместимости сериализованного command body, но обрабатывает работу из `CatalogCacheWarmRequestStore`: generation, refresh marker и bounded integer title IDs. Job использует `ShouldBeUniqueUntilProcessing`, versioned недельный Redis unique lock, versioned overlap lock, timeout 300 s, tries 3 и backoff 30/120/300. Absolute `retryUntil` отсутствует, поэтому долго ожидающая worker job не истекает до `handle()`; реальные ошибки по-прежнему ограничены тремя attempts. Intent подтверждается только после успешной data/HTTP warm пачки; новый generation создаёт tail job. Старые `cache-warm` envelopes с уже истёкшим `retryUntil` не могут дойти до no-op и поэтому остаются в изолированной очереди без автоматического retry/forget.
- Horizon не установлен: текущая инфраструктура не подтверждает отдельный поддерживаемый non-cluster Redis deployment и авторизованный UI. При его добавлении нельзя использовать зарезервированный Horizon connection для cache/session/locks.

## Jobs

- `App\Jobs\RunSeasonvarImport` — внутренний queued wrapper для тяжелого импорта Seasonvar. Публичной командой остается `php artisan seasonvar:import`.
- `RefreshSeasonvarCatalogTitle` — уникальный по `catalog_title_id` targeted job в приоритетной очереди `seasonvar-title-refresh`. Он принимает только ID, читает и валидирует сохранённый URL, использует тот же `SeasonvarImportGroupKey` lock, что page jobs, запускает forced pipeline в пределах настроенного retry window и после успеха объединяет подтверждённые дубли сезонного семейства в открытую карточку. Если тайтл удалён после dispatch, job очищает refresh state и успешно завершается без retry. Redis/cache state содержит только статус, timestamps и import run ID; URL и exception message в UI state/log context не попадают.
- `StartSeasonvarQueuedImport` — короткий coordinator для `/admin/imports`: payload содержит только `seasonvar_import_runs.id`, затем job вызывает тот же `SeasonvarQueuedImportDispatcher`, что CLI `--queued`.
- `SeasonvarGlobalImportRunCoordinator` является общей single-flight границей полного sitemap import для sync CLI/legacy wrapper, admin, queued CLI/cron и retry. Под одним коротким start-lock active lookup охватывает `sync` и `queue` execution modes; sync path заранее резервирует running lifecycle row, а pipeline выполняет её без второй вставки. Повторный глобальный вызов переиспользует active `queued/running` sitemap run и не dispatches page work второй раз. Lock освобождается до provider HTTP и catalog work. Targeted URL/title refresh, inventory и status остаются независимыми. Guard предотвращает новые пересечения, но не изменяет и не завершает уже существующие production runs.
- Coordinator ограничен 3 attempts, timeout 900 секунд и backoff 60/300/900; `ShouldBeUnique` держит lock на ID run. Page/finalizer jobs ограничены absolute retry window, согласованным с claim lease.
- Контракт повторов задан на каждом из восьми importer jobs. `RunSeasonvarImport`, `StartSeasonvarQueuedImport` и `WakeSeasonvarImportFinalizers` ограничены тремя attempts, имеют соответственно timeout `900/900/120`, backoff `60,300,900` / `60,300,900` / `30,120,300`, unique lock и безопасный `failed()` boundary. `ImportSeasonvarSourcePage`, `PrepareSeasonvarImportTitlePage`, оба finalizer и `RefreshSeasonvarCatalogTitle` используют `tries=0` только вместе с абсолютным `retryUntil`, timeout 900, явным backoff и `failed()`; worker `--tries=0` не отменяет их job-level deadline. Page job намеренно не реализует `ShouldBeUnique`, потому что право на конкретную доставку уже задаёт проверяемый claim token; preparation, finalizers и targeted refresh остаются unique по стабильным ID.
- Максимальный importer timeout 900 секунд обязан оставаться меньше `retry_after=1200`: worker успевает завершить или потерять попытку до повторной доставки одного queue envelope. Изменение timeout, claim/retry windows, worker flags или `retry_after` требует одновременной проверки этой границы и regression-контракта `SeasonvarQueueJobContractTest`.
- `SeasonvarImportFailureClassifier` разделяет temporary и permanent failures. Первые повторяют coordinator, вторые сразу фиксируют `failed`; queue logs и persisted error fields проходят `SeasonvarImportErrorSanitizer`.
- Queued finalizer выбирает `licensed_media` только по наступившему `next_check_at`, в стабильном `id`-порядке проверяет не более `SEASONVAR_MEDIA_CHECK_MAX_PER_CYCLE` строк и применяет health state вне catalog transaction. Default cap `20` совместим с worker timeout при трёх 10-секундных попытках; `SEASONVAR_MEDIA_CHECK_CHUNK_SIZE` ограничивает только размер чтения. После batch finalizer пересобирает рекомендации и stats snapshot; permanent source error не делает сам finalizer job failed.
- Перед внешним выбором и catalog-wide derived work finalizer выполняет bounded local metadata-backfill из сохранённых snapshots. Page/title limits не дают накопившемуся version backlog вытеснить остальные стадии одного 900-секундного job.
- Recommendation rebuild ограничивает предварительный candidate pool через `SEASONVAR_RECOMMENDATION_CANDIDATE_LIMIT` и `SEASONVAR_RECOMMENDATION_CANDIDATE_SCAN_PER_FEATURE`; широкие слабые признаки учитываются только после отбора. Это удерживает полный rebuild внутри worker memory/timeout boundary.
- Production template запускает PHP с `memory_limit=256M`, но просит Laravel завершить worker при `--memory=192`; `Restart=always` сразу создаёт свежий процесс. `--memory` сам по себе не повышает PHP hard limit и не заменяет эту пару настроек.
- Group/global finalizers используют `ShouldBeUniqueUntilProcessing`: completion event dispatches следующий wake-up только после освобождения unique key перед обработкой. Если siblings, live claims или groups ещё не terminal, job обновляет heartbeat и завершается без self-release, поэтому ожидание не расходует attempts. Дополнительный Redis lock `seasonvar-import-finalizer` сериализует catalog-wide finalization между runs, имеет TTL `job timeout + 300` и освобождается в `finally`; только реальная конкуренция за apply/global lock использует delayed release. Run state и claims проверяются повторно уже под lock.
- Уникальный `WakeSeasonvarImportFinalizers` scheduled каждые 10 минут через `redis-locks`, `withoutOverlapping` и `onOneServer`. Он bounded-пачкой повторяет сигналы для counter-ready групп, active global runs и active групп старше `max(retry window, claim window) + 300s`; default `SEASONVAR_QUEUE_FINALIZER_WATCHDOG_BATCH_SIZE=250`. Watchdog сам не меняет group/page state, не очищает и не retry-ит failed jobs.
- Получив stale-сигнал, group finalizer в короткой transaction повторно проверяет active run, возраст строк и отсутствие live claim. Свежая работа и живой claim остаются без изменений. Старая `queued/preparing` строка без возможного queue retry получает allowlisted `preparation_deadline_exceeded`; подготовленные siblings по-прежнему применяются и дают `partial`. Нулевой или структурно неполный page set становится `failed` с `empty_page_set`/`page_set_mismatch`. `no_prepared_pages` и `finalizer_deadline_exceeded` закрывают остальные terminal failure boundaries; русское сообщение выводится из enum и не содержит URL, payload или exception text.
- Bounded watchdog query поддерживает индекс `(status, updated_at, id)`. Nullable `terminal_reason_code` не участвует в eligibility query и индекс ему не нужен; он хранит только стабильные internal codes, а не переведённый label или provider diagnostic.
- Если watchdog исчерпал attempts, его `failed()` пишет только стабильный job code и класс исключения. Exception message, payload, URL, token и другие private values в этот лог не попадают; следующая scheduled доставка остаётся штатным механизмом восстановления сигналов.
- Run states: `queued -> running -> completed|partial|failed`; admin cancel и штатная signal-остановка sync CLI дают `cancelled`. `SIGINT`/`SIGTERM` регистрируются Laravel signal registry: текущая страница завершает свой HTTP/DB boundary, следующая не начинается, счётчики отражают только реально выбранные страницы, а events хранят stop phase без URL и exception details. Heartbeat обновляется coordinator/page/finalizer шагами, а admin service может закрыть `running` run с просроченным heartbeat, если живых claims больше нет. Порог — `SEASONVAR_QUEUE_STALE_AFTER_MINUTES`.
- Retry из admin UI разрешён только для `partial/failed` и создаёт новую audit-строку. Для старых Laravel failed jobs остаётся `php artisan queue:failed`/`queue:retry <id>`; перед retry нужно сверить run state и claims.
- Job принимает только scalar-параметры: URL-аргумент, `force` и `discover`. Модели или большие структуры данных в очередь не передаются.
- Job не поддерживает `forever`-режим: queued import всегда выполняет ограниченный запуск, чтобы worker не занимался бесконечным циклом.
- Job реализует `ShouldBeUnique`, использует уникальный ключ `seasonvar-import` и дополнительно берет тот же lock из configured critical store `redis-locks`, что CLI-команда. Если импорт уже идет, job освобождает себя обратно в очередь через 300 секунд; domain cache в координации не участвует.
- Политика выполнения job: `tries=3`, `timeout=900`, backoff `[60, 300, 900]`, `uniqueFor=3600`.
- Ошибки job логируются через `failed()` с URL-аргументом и флагами запуска; сама pipeline уже помечает созданный `SeasonvarImportRun` как failed.
- Если задан `SEASONVAR_IMPORT_FAILURE_MAIL_TO`, `RunSeasonvarImport::failed()` отправляет queued on-demand notification `SeasonvarImportFailed` на email получателя. Notification использует queue из `NOTIFICATIONS_MAIL_QUEUE`.

## Безопасная сводка failed jobs

`php artisan app:deployment-check --json` читает `failed_jobs` bounded chunks, но получает из SQL только allowlisted display name, первые 256 байт exception для стабильной reason-категории, queue category и failed time. Он выводит total и число job/category/age/reason buckets. Payload, полный exception text, URL, аргументы и токены не materialize в health path и не включаются в результат. Наличие строк помечается warning для обязательной ручной disposition; команда не выполняет retry, forget или очистку.

`php artisan app:failed-job-audit --json --samples=1` — отдельный forensic path. Он рассматривает только два exact allowlisted finalizer class, забирает из SQL не более 16 KiB payload и 256 байт exception, проверяет JSON display/command name, serialized root и один direct positive integer `groupId`/`importRunId` собственным bounded parser без создания PHP object. Затем 200-row chunks связываются grouped queries с current group/run status, live claims и active child groups. JSON содержит только counts, stable codes и bounded samples из failed ID/type/target ID/reason/age/state/disposition; queue UUID, command body, exception, URL, token и provider data отсутствуют.

Disposition meanings:

- `forget_candidate` — target отсутствует или уже terminal; удаление возможно только отдельным recorded operator decision по конкретным IDs;
- `retain` — у target есть текущая active work или live claim;
- `canonical_signal_candidate` — target уже пригоден для современного watchdog/signal path; старый serialized envelope не retry-ится;
- `manual_review` — payload не подтверждён или current parent state противоречив.

Audit всегда возвращает нулевые `retried/forgotten/cleared/dispatched` counters и не имеет mutation options. Перед любым отдельным `queue:forget <id>` сохранить JSON evidence, записать решение и повторить count/audit. `queue:retry all`, массовый forget и `queue:clear` не входят в recovery flow.
